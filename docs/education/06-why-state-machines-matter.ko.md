# nano-backend.ai에서 상태 머신이 중요한 이유

## 누구를 위한 문서인가
이 문서는 `queued`, `preparing`, `running`, `failed` 같은 상태 이름을 자꾸 보는데, 왜 플랫폼이 이를 명시적으로 모델링하는지 궁금한 백엔드·인프라 엔지니어를 위한 문서입니다.

## 왜 중요한가
run 상태 머신은 보기 좋은 문서 장식이 아닙니다. 이것은 플랫폼을 다음과 같이 만들어 주는 핵심 구조입니다.
- 디버깅 가능하게 만들고,
- 자동화 가능하게 만들고,
- 재시도를 안전하게 만들고,
- 운영 상태를 읽기 쉽게 만듭니다.

명시적인 상태 머신이 없으면 시스템은 결국 이런 모호한 이야기로 무너집니다.
- "뭔가 시작된 것 같다"
- "뭔가 실패한 것 같다"
- "일단 다시 돌려 볼까"

이 정도 정보로는 사람도, 에이전트도 제대로 판단할 수 없습니다.

## 큰 그림 직관
이렇게 생각하면 이해가 쉽습니다.

> 상태 머신은 **run이 지금 어디에 있는지**와 **다음에 어떤 전이가 허용되는지**에 대한 플랫폼의 약속입니다.

즉 상태 머신은 비공식적인 과정을 통제된 워크플로로 바꿔 줍니다.

조금 더 실무적으로 말하면, 상태 머신은 두 질문에 답합니다.
1. 지금 무슨 일이 일어나고 있는가?
2. 다음에 허용되는 합법적인 단계는 무엇인가?

이 질문이 단순해 보여도, 실제로는 큐잉·실행·실패 처리·재시도를 추측이 아니라 규칙으로 다루게 해 줍니다.

## nano-backend.ai의 상태 모델
MVP 상태 머신은 의도적으로 작게 잡혀 있습니다.

```text
queued -> preparing -> running -> succeeded
              |            |
              +-- failed   +-- failed / cancelled
```

그리고 각 상태의 의미는 구체적입니다.

| 상태 | 의미 |
|---|---|
| `queued` | run은 접수되었지만 아직 GPU를 기다리는 중 |
| `preparing` | 이미지, 모델, 데이터셋, mount 등 실행 환경을 준비하는 중 |
| `running` | trainer 프로세스가 실제로 시작된 상태 |
| `succeeded` | trainer가 성공적으로 종료했고 출력이 수집된 상태 |
| `failed` | run이 비정상적으로 종료된 상태 |
| `cancelled` | 사용자 또는 에이전트가 의도적으로 중단한 상태 |

중요한 점은 이름 자체가 아니라, 각 상태가 운영적으로 분명한 의미를 가진다는 것입니다.

## 왜 하나의 일반적인 status로는 부족한가
만약 플랫폼에 다음 상태만 있다고 해봅시다.
- `pending`
- `running`
- `done`
- `failed`

겉보기에 단순해 보이지만, 실제로는 너무 많은 의미를 숨깁니다.

예를 들어 다음처럼 전혀 다른 상황이 모두 `failed` 하나로 뭉개집니다.
- image pull 실패,
- model download 실패,
- trainer crash,
- GPU OOM,
- 의도적인 cancel.

이렇게 뭉개 버리면 에이전트는 더 이상 알 수 없습니다.
- 같은 spec을 다시 넣어도 되는지,
- config를 바꿔야 하는지,
- 운영자를 불러야 하는지,
- artifact bundle을 믿어도 되는지.

즉 상태 머신의 목적은 보기 좋은 설계가 아니라, 더 나은 의사결정을 가능하게 하는 데 있습니다.

## 상태가 에이전트의 사고를 어떻게 돕는가
좋은 상태 머신은 에이전트가 해야 할 추측의 양을 줄여 줍니다.

### `queued`
run은 일단 받아들여졌지만 아직 GPU가 배정되지 않은 상태입니다.

이 상태는 에이전트에게 이렇게 말해 줍니다.
- 최소한 초기 검증은 통과했다
- 지금 문제는 config 파싱이 아니라 scheduling이다
- 다시 제출할 때가 아니라 polling할 때다

### `preparing`
플랫폼이 실행 환경을 실제로 만들고 있는 단계입니다.

예를 들면:
- container image pull,
- base model 다운로드 또는 위치 확인,
- dataset staging,
- mount 및 runtime 설정 검증.

이 상태는 에이전트에게 이렇게 말해 줍니다.
- 학습은 아직 진짜로 시작되지 않았다
- 여기서 나는 실패는 환경 문제일 가능성이 높다
- 같은 spec 재시도가 유효할 수 있다

### `running`
이 상태가 중요한 경계입니다.

trainer 프로세스가 시작된 순간부터 run은 단순한 scheduling 객체가 아니라, 실제 실행 예산을 소비하는 대상이 됩니다.

이 상태는 에이전트에게 이렇게 말해 줍니다.
- 이제 로그가 중요하다
- 이제 자원 크기 산정이 중요하다
- 여기서의 실패는 실험 설정이나 trainer와 더 직접적으로 연결될 가능성이 높다

### 종료 상태: `succeeded`, `failed`, `cancelled`
이 상태들은 run 생애주기를 닫습니다.

종료 상태라는 뜻은:
- 더 이상 앞으로 갈 상태 전이가 없고,
- artifact를 검사할 수 있으며,
- 새로운 시도는 새 run으로 만들어야 한다는 뜻입니다.

이 점이 중요한 이유는 run의 정체성을 깨끗하게 유지하기 위해서입니다. run은 계속 재활용되는 슬롯이 아니라, 한 번 발생한 불변 이벤트여야 합니다.

## 전이 경계가 왜 중요한가
이 시스템에서 가장 교육적으로 중요한 경계는 다음입니다.

> `preparing` -> `running`

이 선은 다음을 나눕니다.
- 환경/준비 문제
- 실제 실행 문제

그래서 이 경계가 재시도 정책에서 핵심입니다.

예를 들면:
- `preparing` 단계의 `image_pull_failed` -> 같은 spec이 여전히 맞을 수 있음
- `running` 단계의 `oom` -> 같은 spec은 대체로 너무 공격적임

이 경계가 없으면 모든 실패가 비슷하게 보이고, 무작정 재시도하다가 GPU 시간을 낭비하게 됩니다.

## 상태 머신과 artifact 신뢰도
상태 머신은 결과 artifact를 얼마나 믿어야 하는지도 알려 줍니다.

예를 들면:
- `succeeded` run의 완전한 출력은 강한 증거입니다.
- `preparing`에서 끝난 `failed` run은 trainer 출력이 거의 없을 가능성이 큽니다.
- `running`에서 실패한 run은 부분 로그와 유용한 단서를 남길 수 있습니다.
- `cancelled` run도 성능이나 비용 분석에는 충분한 출력이 남아 있을 수 있습니다.

즉 상태는 실행 제어만 위한 것이 아니라, 남은 증거를 어떻게 해석해야 하는지까지 알려 줍니다.

## 흔한 오해
### 오해 1: "상태 머신은 백엔드 관료주의다"
아닙니다. 상태 머신은 자동화를 가능하게 하는 핵심 primitive입니다. 에이전트는 안전한 다음 액션 결정을 위해 명시적 상태가 필요합니다.

### 오해 2: "나중에 로그를 보면 다 복원할 수 있다"
로그는 중요하지만, 구조화된 상태를 대신할 수는 없습니다. 로그만으로 의미를 복원하려 하면 더 느리고, 더 불안정하고, 자동화하기도 어렵습니다.

### 오해 3: "상태 수가 적을수록 무조건 좋다"
상태 수를 줄이는 것이 좋은 것은 운영적 의미를 보존할 때만 그렇습니다. 너무 압축하면 오히려 모호성이 커집니다.

## 실전 디버깅 체크리스트
run이 기대와 다르게 보일 때는 다음 순서로 보세요.

1. 지금 run은 어떤 상태인가?
2. 그 상태는 플랫폼이 현재 해야 할 일과 맞는가?
3. 실패했다면 `running` 전에 실패했는가, 후에 실패했는가?
4. `failure_reason`이 상태 전이와 맞게 기록되었는가?
5. 남아 있는 artifact는 그 종료 상태와 일관적인가?
6. 다음 액션은 poll, retry, spec 수정, log inspection 중 무엇인가?

## 간단한 운영 예시
두 run이 있다고 해봅시다.

- Run A는 `queued`에 머물러 있음
- Run B는 `running`에서 `oom`으로 실패함

이 둘은 전혀 다른 대응이 필요합니다.

Run A에서는:
- scheduler 압력을 보고,
- GPU 점유 상태를 보고,
- 아직 spec을 고칠 단계는 아닙니다.

Run B에서는:
- batch size, sequence length, model size를 보고,
- 로그와 메트릭을 확인하고,
- scheduler를 탓하면 안 됩니다.

상태 머신의 목적은 바로 이런 구분을 분명하게 만드는 데 있습니다.

## 핵심 요약
- 상태 머신은 run 생애주기에 대한 플랫폼의 계약입니다.
- 명시적 상태가 있어야 자동화, 재시도, 디버깅이 안전해집니다.
- 특히 `preparing` -> `running` 경계는 환경 실패와 실행 실패를 가르기 때문에 중요합니다.
- 종료 상태는 run 정체성을 불변으로 만들고 artifact 해석을 가능하게 합니다.
- 에이전트가 상태 전이를 신뢰할 수 없으면 결국 추측에 의존하게 되고, 추측은 비쌉니다.
