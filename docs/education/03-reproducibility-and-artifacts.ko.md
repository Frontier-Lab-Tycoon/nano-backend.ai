# 재현성과 아티팩트 규율

## 누구를 위한 문서인가
이 문서는 nano-backend.ai에서 run 기록과 아티팩트 구조를 설계하거나 운영하는 사람을 위한 문서입니다.

에이전트가 실험을 자동으로 제출하고 비교하는 워크플로를 만든다면, 재현성은 있으면 좋은 옵션이 아니라 자동화를 안전하게 만드는 핵심 조건입니다.

## 재현성은 운영 요구사항이다
ML에서는 재현성을 흔히 연구 윤리나 과학적 엄밀성의 문제로 설명합니다. 그 말도 맞습니다. 하지만 에이전트 기반 플랫폼에서는 그보다 더 실무적인 의미가 있습니다.

에이전트는 다음 질문에 사람 기억 없이 답할 수 있어야 합니다.
- run A와 run B 사이에 무엇이 달라졌는가?
- 같은 run을 정확히 다시 실행할 수 있는가?
- 이 어댑터를 다음 run의 입력으로 써도 되는가?
- 이 아티팩트를 만든 코드, 설정, 이미지는 무엇인가?

답이 채팅 기록이나 임시 셸 히스토리에만 남아 있다면, 그 시스템은 자동화에 충분히 재현 가능하다고 보기 어렵습니다.

## 재현성의 세 층위
### 1. 입력 재현성
다음이 같아야 합니다.
- 베이스 모델,
- 데이터셋,
- 프리셋,
- override,
- 출력 계약.

### 2. 환경 재현성
다음이 같아야 합니다.
- 컨테이너 이미지,
- 패키지 버전,
- 런타임 가정,
- 마운트 구조,
- 필요하다면 캐시 동작.

### 3. 계보(provenance) 재현성
다음이 끊기지 않고 이어져야 합니다.
- git commit,
- run ID,
- preset/config,
- artifact bundle,
- lineage metadata.

좋은 시스템은 이 세 가지를 함께 강제합니다. 한 층이라도 약하면 run에 대한 신뢰도 약해집니다.

## 무엇을 진실의 원천으로 볼 것인가
nano-backend.ai에서 진실의 원천은 파일 하나가 아닙니다. 서로를 보강하는 작은 기록 묶음입니다.

- `spec.yaml`: 사용자가 제출한 내용
- `resolved_config.yaml`: 시스템이 실제로 실행한 내용
- run metadata: `run_id`, 시간 정보, 상태, `failure_reason`
- lineage metadata: 예를 들어 `git_sha`, 이슈/PR/스레드 참조
- artifact bundle: 로그, 메트릭, 리포트, 어댑터, 선택적 merged 결과물

이 묶음이 함께 실험 원장 역할을 합니다.

## `spec.yaml`과 `resolved_config.yaml`의 차이
이 차이는 반드시 명확히 이해해야 합니다.

### `spec.yaml`
사용자의 의도를 기록합니다.

즉 다음 질문에 답합니다.
- 사용자가 무엇을 요청했는가?
- 어떤 입력과 출력이 선언되었는가?
- 어떤 프로젝트/lineage 정보가 붙었는가?

### `resolved_config.yaml`
실행 현실을 기록합니다.

즉 다음 질문에 답합니다.
- 프리셋 기본값이 어떻게 채워졌는가?
- 어떤 override가 실제로 적용되었는가?
- 트레이너가 실제로 본 설정은 무엇인가?

둘 다 필요합니다.
`spec.yaml`만 있으면 원래 의도만 남고 실제 실행값이 가려집니다.
`resolved_config.yaml`만 있으면 실행값은 보여도 원래 요청 의도를 추적하기 어렵습니다.

## 아티팩트 번들 계약
모든 run은 예측 가능한 번들을 만들어야 합니다.

| 아티팩트 | 필요한 이유 |
|---|---|
| `spec.yaml` | 원래 제출 내용을 보존 |
| `resolved_config.yaml` | 실제 실행 설정을 보존 |
| `stdout.log` | 트레이너 전체 출력 기록 |
| `stderr.log` | 에러와 경고 확인 |
| `metrics.json` | 비교 가능한 구조화 메트릭 |
| `report.md` | 사람이 빠르게 읽을 수 있는 요약 |
| `adapter/` | 핵심 학습 결과물 |
| `merged/` | 선택적 merged 결과물 |

이 번들은 단순한 출력 폴더가 아니라 증거 패키지입니다.

## 어떤 run이 불완전한가
파일이 몇 개 생겼다고 해서 곧바로 신뢰할 수 있는 run이 되는 것은 아닙니다.

예를 들면:
- 어댑터는 있는데 로그가 없거나,
- 메트릭이 stdout 안의 텍스트로만 남아 있거나,
- `resolved_config.yaml`이 없거나,
- lineage에 `git_sha`가 빠져 있거나,
- 아티팩트 디렉터리가 덮어써졌다면,

그 run은 일부 결과물은 유용할 수 있어도 깔끔한 원장 항목으로 보기는 어렵습니다.

## 실전 예시: 두 run 비교하기
예를 들어 run A와 run B의 차이가 `learning_rate` 하나뿐이라고 합시다.

좋은 비교는 대체로 이렇게 진행됩니다.
1. `resolved_config.yaml`을 diff한다.
2. 베이스 모델, 이미지, 데이터셋, preset이 그대로인지 확인한다.
3. `metrics.json`을 본다.
4. `report.md`로 빠르게 요약을 확인한다.
5. 구조화 데이터가 이상할 때만 로그를 자세히 본다.

이 방식이 자유 서술 메모를 비교하는 것보다 훨씬 믿을 만합니다.

## 실전 예시: 실험 이어붙이기
이번에는 run A의 어댑터를 run B의 시작점으로 쓰고 싶다고 해봅시다.

안전하게 그렇게 하려면 다음을 알아야 합니다.
- 이 어댑터가 어떤 베이스 모델에서 나왔는가?
- 어떤 preset과 config가 이를 만들었는가?
- run이 정상 종료했는가?
- artifact bundle이 신뢰할 만큼 완전한가?

바로 그래서 lineage와 아티팩트 규율이 중요합니다. 증거 없이 재사용하면 결국 추측이 됩니다.

## 재현성을 깨뜨리는 흔한 실수
대표적인 문제는 다음과 같습니다.
- `latest` 같은 가변 이미지 태그 사용,
- 아티팩트 디렉터리 덮어쓰기,
- 설정 스냅샷 누락,
- 메트릭을 비구조화 로그에만 남기기,
- `git_sha` 누락,
- 불명확한 lineage,
- 부분 출력이 있는데 상태 의미가 명확하지 않은 경우.

이건 문서 작업이 귀찮아서 생기는 문제가 아닙니다. 실험 신뢰도를 직접 깎아먹는 문제입니다.

## 디버깅 체크리스트: 이 run을 믿어도 되는가?
이 run을 자동화의 근거로 써도 되는지 판단하려면 먼저 물어보세요.

1. `spec.yaml`과 `resolved_config.yaml`이 모두 존재하는가?
2. 아티팩트 디렉터리가 이 `run_id`에만 대응되는가?
3. 이미지가 불변 참조로 고정되어 있는가?
4. lineage에 `git_sha`가 기록되어 있는가?
5. 메트릭이 구조화된 JSON으로 남아 있는가?
6. 로그가 최종 상태를 설명해 주는가?
7. 어댑터가 있다면, 그것이 어떤 설정과 코드에서 나왔는지 설명할 수 있는가?

이 질문들에 여러 개가 "아니오"라면, 그 run은 흥미로운 힌트를 줄 수는 있어도 자동화의 강한 기반으로 삼기에는 약합니다.

## 핵심 요약
- 재현성은 논문용 미덕이 아니라 에이전트 자동화를 위한 운영 요구사항입니다.
- `spec.yaml`은 의도를 기록하고, `resolved_config.yaml`은 실행 현실을 기록합니다.
- 아티팩트 번들은 출력 모음이 아니라 증거 패키지입니다.
- 계보와 증거가 빈약한 run은 흥미로운 이야기일 수는 있어도, 신뢰할 수 있는 실험이라고 보기는 어렵습니다.
