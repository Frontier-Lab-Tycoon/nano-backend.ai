# 재현성성(Reproducibility)과 아티팩트 규율

## 재현성성은 보너스 기능이 아닙니다

ML 연구에서 결과를 재현할 수 없다면 그 결과를 믿을 수 없습니다. 에이전트 정동 워클로우에서는 재현성성이 더욱 엄격합니다: 에이전트는 인간의 기억으로 공백을 때우지 않고 재제출, 비교, 실험 찹을 돌려야 합니다.

## 재현성성의 세 계층

1. **입력 재현성성**: 동일한 모델, 동일한 데이터, 동일한 하이퍼파라미터.
2. **환경 재현성성**: 동일한 Docker 이미지, 동일한 라이브러리 버전, 동일한 CUDA 드라이버.
3. **출처(Provenance) 재현성성**: git commit에서 run ID로, 그리고 아티팩트 해시(hash)로 이어지는 완전한 채입입니다.

nano-backend.ai는 설계상 모든 세 계층을 강제합니다.

## 아티팩트 계약

모든 런은 표준화된 번들을 생성해야 합니다. 이것은 "가져옴 돼서 좋은 것"이 아니라 대장(ledger)입니다.

| 파일 | 목적 |
|------|------|
| `spec.yaml` | 정확히 무엇을 제출했는지. 모호함 없음. |
| `resolved_config.yaml` | 실제로 실행된 것 (기본값 + 오버라이드 병합). |
| `stdout.log` / `stderr.log` | 완전한 후학습 전사(transcript). |
| `metrics.json` | 후속 비교를 위한 구조화된 데이터. |
| `report.md` | 빠른 검토를 위한 인간 읽기 쉬운 요약. |
| `adapter/` | 실제 학습 결과물. |

`spec.yaml`과 `resolved_config.yaml`가 없으면 런은 **불완전**하고 대장 항목으로 신뢰할 수 없습니다.

## 에이전트에겐 왜 중요한가

에이전트는 다음을 할 수 있어야 합니다:
- 런 A와 런 B를 `resolved_config.yaml`을 diff해 비교합니다.
- 런 A의 `adapter/`를 가져와 서 2차 후학습의 `base_model`로 사용합니다.
- 근거 실험의 `spec.yaml`을 복사하고 한 가지 오버라이드만 바꿔서 재실행합니다.

아티팩트 규율이 없으면 에이전트는 추측하거나 인간에 물어봐야 합니다. 그러면 자동화가 꺾입니다.

## 실전 규칙

- 아티팩트 디렉터리를 덮어씌우지 마세요. Run ID는 불변입니다.
- Preset의 이미지 태그 `latest`를 믿지 마세요. Digest를 고정하세요.
- `lineage`에 `git_sha`를 항상 기록해서 코드 상태를 복원할 수 있게 하세요.
- 메트릭을 로그 텍스트가 아니라 구조화된 데이터로 다룬 세요. 정규표현식으로 로그를 파싱하는 것은 취약합니다.

## 요약

아티팩트 번들은 "출력"이 아닙니다. 그것은 **증거(evidence)**입니다. 완전한 아티팩트 번들이 없는 런은 이야기(anecdote)일 뿐, 실험이 아닙니다.
