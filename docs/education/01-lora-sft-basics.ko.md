# LoRA / SFT: 실전에서 꼭 알아야 할 것

## SFT는 무엇인가

Supervised Fine-Tuning (SFT, 지도 미세조정)은 사전 학습된 베이스 모델을 가져와, 작고 레이블이 붙은 데이터셋으로 추가 학습시키는 것이다. 베이스 모델은 이미 언어를 이해하고 있다. SFT는 그 모델에게 특정 작업의 지시 형식을 가르치는 과정이다.

## LoRA는 무엇인가

Low-Rank Adaptation (LoRA, 저차원 적응)은 파라미터 효율적인 미세조정 방법이다. 수십억 개 파라미터를 모두 업데이트하는 대신, 어텐션과 피드포워드 레이어에 작고 학습 가능한 어댑터 행렬을 주입한다.

수학은 단순하다:
- 일반적인 업데이트: `W' = W + ΔW`
- LoRA는 근사한다: `ΔW = A × B`, 여기서 `A`와 `B`는 rank `r`을 가진 작은 행렬
- `W`는 얼리고(freeze), `A`와 `B`만 학습한다

이것이 중요한 이유는 `A`와 `B`의 총합이 50MB일 수 있는데, 베이스 모델은 7GB 이상이기 때문이다. 메모리를 절약하고, 디스크 공간을 절약하고, 반복 실험 속도를 높인다.

## 왜 nano-backend.ai의 MergeOwl에서 중요한가

당신은 RTX 3090 2장(각 24GB VRAM)을 탑재한 홈서버에서 실행한다. 7B 모델의 전체 미세조정은 40GB 이상의 VRAM과 모델 병렬화가 가능한 다중 GPU 설정이 필요하다. LoRA는 단일 3090에서도 학습이 가능하게 한다.

| | 전체 미세조정 | LoRA |
|---|---|---|
| 업데이트되는 파라미터 | 100% | ~0.1–1% |
| 필요한 VRAM | 40GB 이상 | 16–22GB |
| 체크포인트 크기 | 7GB 이상 | 10–200MB |
| 실험 반복 속도 | 실험당 수 시간 | 큐잉은 분 단위, 학습은 시간 단위 |
| 어댑터 교체 | 전체 모델 재로드 | 50MB 파일 교체, 베이스는 VRAM에 그대로 |

nano-backend.ai의 프리셋 시스템은 베이스 모델 이미지를 입력으로 받고 `adapter/` 아티팩트를 출력으로 낸다. 이 설계는 LoRA를 전제로 한다. 전체 미세조정을 시도하면 첫 번째 포워드 패스 전에 컨테이너가 OOM으로 죽는다.

## 멘탈 모델: LoRA가 시스템 안에서 어떻게 작동하는가

베이스 모델을 고정된 엔진으로 생각하라. LoRA 어댑터는 특정 레이어에 볼트로 고정된 튜닝 가능한 칼브레이터다. 추론 시 플랫폼은 어댑터를 베이스 가중치에 다시 병합(merge)해서, 마치 전체 미세조정한 것처럼 동작하게 한다. 하지만 학습 중에는 칼브레이터만 바뀐다.

핵심 함의: 하나의 베이스 모델을 GPU 메모리에 유지한 채 어댑터를 수 초 안에 교체할 수 있다. 그래서 MergeOwl은 실험 A → 평가 → 실험 B(실험 A의 어댑터를 베이스로 사용)라는 체인을 7GB 체크포인트를 재로드하지 않고도 실행할 수 있다.

## 실제로 중요한 하이퍼파라미터

이것들은 `POST /runs`에서 오버라이드할 수 있는 노브다. 하나라도 틀리면 학습이 조용히 실패하거나 폭발한다.

| 파라미터 | 제어 대상 | 일반적인 범위 | 위험 구간 |
|---|---|---|---|
| `learning_rate` | 어댑터가 얼마나 빠르게 업데이트되는가 | 1e-4 ~ 5e-4 | >1e-3: loss NaN; <1e-5: 학습 안 됨 |
| `num_epochs` | 데이터셋을 몇 번 도는가 | 1–5 | >10: 소규모 데이터에서 과적합; 0: 학습 없음 |
| `lora_r` | 어댑터 행렬의 rank | 8, 16, 32, 64 | 너무 높음: OOM; 너무 낮음: 과소적합 |
| `lora_alpha` | 어댑터 출력에 적용되는 스케일링 계수 | 보통 `2 × r` | `r`과 불일치 시 기울기 불안정 |
| `max_seq_length` | 학습 중 모델이 보는 최장 시퀀스 길이 | 512, 1024, 2048 | 너무 높음: OOM; 너무 낮음: 데이터가 잘림 |
| `micro_batch_size` | 한 번의 forward/backward에 쓰이는 샘플 수 | 1–4 | 너무 높음: OOM; 너무 낮음: 수렴 느림 |

3090에서의 경험 법칙: `lora_r=64`이고 `max_seq_length=2048`이면 `micro_batch_size=1`이어야 한다. `micro_batch_size=2`를 원하면 `lora_r`을 32로 내리거나 시퀀스 길이를 줄여라.

## nano-backend.ai에서의 시스템 동작

`axolotl-lora-sft` 프리셋으로 런을 제출하면:

1. **Queue**: 스케줄러가 현재 부하를 보고 GPU 0 또는 1에 할당한다.
2. **Preparing**: 컨테이너가 베이스 모델 이미지를 pull하고, 데이터셋이 없으면 다운로드하고, 캐시를 워밍업한다.
3. **Running**: 트레이너가 베이스 모델을 VRAM에 로드하고, LoRA 어댑터를 부착한 뒤 forward/backward 패스를 실행한다.
4. **Terminal**: 트레이너가 `adapter/` 디렉터리를 저장하고, 메트릭을 로깅하고, `report.md`를 작성한다.
5. **Artifact verify**: 플랫폼이 `adapter/`가 존재하고 `metrics.json`이 유효한 JSON인지 확인한다.

아티팩트 번들 안의 `resolved_config.yaml`은 실제 사용된 값을 기록한다. 재현이나 비교를 원하면 `resolved_config.yaml` 파일들을 diff하라. 제출했던 값을 기억에 의존하지 마라.

## 마주할 실패 모드

| 증상 | 근본 원인 | 먼저 확인할 것 |
|---|---|---|
| 컨테이너가 즉시 종료 코드 137로 종료 | 모델 로드 중 CUDA OOM | `lora_r`이 너무 높거나 `max_seq_length`가 너무 김 |
| 10 스텝 내에 loss가 `NaN`이 됨 | 학습률 너무 높음 | `resolved_config.yaml`의 `learning_rate` |
| 초기값 근처에서 loss가 평평함 | 학습률 너무 낮거나 rank 너무 작음 | `learning_rate`와 `lora_r` |
| 학습은 완료됐는데 어댑터가 0바이트 | 컨테이너 내부 디스크 가득 참 또는 저장 경로 오류 | 컨테이너에서 `df -h`, 아티팩트 마운트 확인 |
| 메트릭이 N 스텝마다 스파이크를 보임 | 그래디언트 누적 설정 오류 | `gradient_accumulation_steps` 대 `micro_batch_size` |
| 학습 후 모델 출력이 이상함 | 추론 시 어댑터 병합이 제대로 안 됨 | 프리셋 설정의 `merge_weights` 플래그 확인 |

## 디버깅 체크리스트

플랫폼을 탓하기 전에, 다음 순서로 확인하라:

1. **`resolved_config.yaml` 확인**: 의도한 대로인가? 프리셋 기본값이 의외일 수 있다.
2. **GPU 메모리 확인**: 컨테이너 내부 또는 호스트에서 `nvidia-smi`. 다른 런이 VRAM을 점유하고 있는가?
3. **`stderr.log` 확인**: `CUDA out of memory`, `RuntimeError`, `NaN`을 검색하라. 첫 번째 에러가 보통 진짜 원인이다.
4. **`metrics.json` 확인**: loss 곡선을 그려보라. 급락 후 NaN = lr 너무 높음. 평평 = lr 너무 낮거나 데이터 문제.
5. **아티팩트 크기 확인**: `adapter/`는 `.bin`이나 `.safetensors` 파일을 포함하고 총합 1MB 이상이어야 한다. 비어 있으면 저장 중 트레이너가 죽은 것이다.
6. **베이스 모델 접근 가능성 확인**: 컨테이너가 Hugging Face Hub에 도달할 수 있는가? 네트워크 타임아웃은 `preparing` 단계에서 행(hang)처럼 보인다.

## 핵심 요약

- SFT는 베이스 모델에게 새로운 작업을 가르친다. LoRA는 이를 컨슈머급 하드웨어에서도 가능하게 한다.
- nano-backend.ai에서는 LoRA를 전제로 한다. Phase 0 프리셋에서 전체 미세조정은 지원하지 않는다.
- 당신의 아티팩트는 `adapter/` 디렉터리이지 병합된 모델이 아니다. 작게 유지하고 버전 관리하라. 실험을 비교할 때는 `resolved_config.yaml`을 diff하라.
- 학습이 실패하면 다음 순서로 확인하라: `resolved_config` → GPU 메모리 → `stderr.log` → `metrics.json` → 아티팩트 크기.
