# 왜 Preset이 Raw Entrypoint보다 나은가?

## 탐낙

에이전트가 후학습 작업을 실행하고 싶다고 생각할 수 있습니다: "내가 쓰고 싶은 쉘 명령어와 Docker 이미지를 주면 안 돼?" 이게 raw entrypoint 접근방식입니다.

유연해 보입니다. 실제로는 재현성 함정(reproducibility trap)입니다.

## Raw entrypoint의 문제

1. **숨겨진 가정**: `python train.py --lr 2e-4`라는 명령은 `train.py`가 존재하고, 특정 패키지가 설치되어 있으며, 환경변수가 설정되어 있을 것을 가정합니다. 이 모든 것이 런 기록에 남지 않습니다.
2. **드리프트(drift)**: 한 달 후 `latest` 태그는 다른 레이어를 가리킵니다. 같은 명령어도 다른 결과를 만듭니다.
3. **에이전트 인지 부하**: 에이전트는 CUDA 버전, Python 경로, Docker 플래그를 이해하는 대신 러닝레이트와 데이터 품질을 이해해야 합니다.
4. **검증 없음**: `--lora_alpha`의 오타를 치면 제출 시점에 검증 오류가 아니라 런타임 오류나 잘못된 동작으로 변합니다.

## Preset이 주는 것

Preset은 검증된 템플릿입니다:
- **Image**: `latest`가 아닌 고정된 digest 또는 변경 불가 태그.
- **Schema**: 허용된 오버라이드 키가 명시적입니다. 잘못된 키는 GPU가 터지기 전에 거부됩니다.
- **Defaults**: 에이전트가 `lora_r`을 지정하지 않으면 preset이 안전한 기본값을 제공합니다.
- **Contract**: 플랫폼은 무엇을 마운트할지, 무엇을 출력으로 기대할지 앞담을 낚니다.

## 비교 예시

| 관심사 | Raw entrypoint | Preset |
|---------|---------------|--------|
| 실험 제출 | 20줄 썜 스크립트 작성 | `axolotl-lora-sft`를 골라 4개의 오버라이드를 설정 |
| GPU 터치기 전 검증 | 없음 | `POST /runs`에서 스키마 체크 |
| 3개월 후 재실행 | 이미지가 아직 있기를 비란 | Preset 레지스트리가 동일한 이미지 + 기본값 보장 |
| 과거 런 조회 | 임의의 쉘 히스토리 읽기 | 구조화된 `resolved_config.yaml` 읽기 |
| 에이전트 머털 모델 | 인프라 엔지니어 | 연구 과학자 |

## 예외를 두는 시점

Phase 0에서는 절대 안 됩니다. Preset 시스템이 안정화되고 50개 이상의 실험을 실행한 후에, `advanced: {custom_image, custom_entrypoint}` 형태의 후문(tambur)을 추가할 수 있습니다. 하지만 기본 경로는 항상 preset-first여야 합니다.

## 요약

Preset은 관주주의가 아닙니다. 그것은 **재현성 원시(primitive)**입니다. "무엇인가 만들어놨다" 라는 이야기를 "시스템이 결정론적으로 실행했다"로 바꿔줍니다.
