# 왜 Raw Entrypoint보다 Preset이 중요한가

## 누구를 위한 문서인가
이 문서는 "에이전트가 Docker 이미지랑 셸 명령만 직접 넘기면 더 유연한 것 아닌가?"라는 질문이 자연스럽게 떠오르는 사람을 위한 문서입니다.

그 생각은 처음엔 자유로워 보입니다. 하지만 에이전트가 실험을 자동으로 제출하고 비교해야 하는 ML 플랫폼에서는 대개 함정에 가깝습니다.

## 핵심 주장
진짜 질문은 "어느 쪽이 더 강력한가"가 아닙니다. 진짜 질문은 **복잡성을 어디에 둘 것인가** 입니다.

- **Raw entrypoint** 는 복잡성을 각 run 안으로 밀어 넣습니다.
- **Preset** 은 그 복잡성을 플랫폼에 모아 두고, 한 번 검증한 뒤, 안전한 조절값만 노출합니다.

Phase 0에서는 이 선택이 거의 항상 preset 쪽으로 기웁니다.

## 이해를 위한 멘탈 모델
두 접근을 이렇게 생각하면 이해가 쉽습니다.

- **Raw entrypoint** = 셸 스크립트를 API처럼 쓰는 방식
- **Preset** = 실험 계약을 API처럼 쓰는 방식

셸 스크립트는 유연하지만 자동화 인터페이스로는 좋지 않습니다. 너무 많은 상태가 암묵적으로 숨어 있기 때문입니다.
Preset은 반대로 run을 읽기 쉬운 형태로 바꿔 줍니다.
- 어떤 이미지가 실행됐는지,
- 어떤 기본값이 적용됐는지,
- 무엇이 override됐는지,
- 어떤 출력이 반드시 나와야 하는지.

## Raw entrypoint에서 실제로 생기는 문제
### 1. 숨겨진 가정
`python train.py --lr 2e-4` 같은 명령 뒤에는 사실 긴 가정 목록이 숨어 있습니다.
- `train.py`가 존재해야 하고,
- Python 경로가 맞아야 하며,
- 패키지가 설치되어 있어야 하고,
- CUDA 버전이 호환되어야 하며,
- 환경변수가 맞게 들어가야 하고,
- 마운트 경로도 정확해야 합니다.

이 가정이 구조적으로 기록되지 않으면, 그 run은 재현 가능하다고 보기 어렵습니다.

### 2. 드리프트
`latest` 태그는 바뀝니다. 패키지 버전도 바뀌고 디렉터리 구조도 바뀝니다. 한 달 전에 통하던 명령이 지금은 조용히 다른 의미를 가질 수 있습니다.

### 3. 비싼 자원을 잡기 전에 검증할 수 없음
`--lora_alpha` 오타 같은 문제는 GPU를 잡은 뒤가 아니라 제출 시점에 막혀야 합니다.

### 4. run 비교가 어려워짐
각 run이 임의의 명령이라면, 두 run 비교는 설정 diff가 아니라 고고학이 됩니다. 구조화된 설정을 비교하는 대신 셸 조각을 해석해야 하기 때문입니다.

### 5. 에이전트의 인지 부하가 잘못 배분됨
에이전트는 다음에 집중해야 합니다.
- 데이터 품질,
- 실험 설계,
- 실패 해석,
- 다음 액션 결정.

Docker 플래그나 셸 quoting 문제에 대부분의 추론 예산을 쓰게 하면 본질에서 멀어집니다.

## Preset이 제공하는 것
Preset은 한 번 검토되고 재사용되는 계약입니다.

보통 다음을 포함합니다.
- 고정된 이미지,
- 이미 정해진 트레이너 엔트리포인트,
- 명시적인 `allowed_overrides`,
- 안전한 기본값,
- 입력/출력 계약,
- 재현 가능한 `resolved_config.yaml`.

즉 사용자는 의미 있는 몇 개의 설정만 바꾸고, 나머지는 플랫폼이 보호합니다.

## 구체적인 비교 예시
Raw entrypoint 세계에서는 흔히 이런 식이 됩니다.

```bash
docker run --rm --gpus all \
  -v /data:/workspace/data \
  -v /cache:/cache \
  my-trainer:latest \
  python train.py --lr 2e-4 --epochs 3 --lora_r 16
```

Preset-first 세계에서는 보통 이렇게 줄어듭니다.

```yaml
preset: axolotl-lora-sft
overrides:
  learning_rate: 2e-4
  num_epochs: 3
  lora_r: 16
```

두 번째 방식이 더 짧고, 더 읽기 쉽고, 검증도 훨씬 쉽습니다. 더 중요한 점은 플랫폼이 "실제로 무엇이 실행됐는지"를 정확히 설명할 수 있게 된다는 것입니다.

## 운영 관점에서 왜 중요한가
Preset은 UX만 좋아지는 장치가 아닙니다.

다음도 함께 좋아집니다.
- 제출 시점 검증,
- 큐 안전성,
- 재현성,
- 운영 지원 용이성,
- 디버깅 품질,
- run 비교,
- 이후 자동화 확장성.

플랫폼이 계약을 이해하고 있어야, 컨테이너가 뜨기 전에도 더 좋은 판단을 할 수 있습니다.

## 디버깅 체크리스트
Preset 기반 run이 기대와 다르게 동작하면 다음 순서로 보세요.

1. 제출한 override 키가 preset schema에 존재하는가?
2. `resolved_config.yaml`이 사용자의 의도를 제대로 반영했는가?
3. 실패 시점이 `preparing`인가 `running`인가?
4. preset이 예상한 이미지와 트레이너 버전을 가리키는가?
5. 필수 아티팩트가 기대한 출력 경로에 생성됐는가?

실전 규칙 하나만 기억하면 됩니다. 컨테이너 내부를 뒤지기 전에 먼저 **실제로 어떤 계약이 실행됐는지** 확인하세요.

## 예외 경로는 언제 열어야 하나
기본값으로는 Phase 0에서 열지 않는 편이 맞습니다.

커스텀 이미지나 커스텀 엔트리포인트 지원은 나중에 필요할 수 있습니다. 다만 그 전제는 분명합니다.
- preset 커버리지가 충분히 넓어졌고,
- 실패 처리 패턴이 충분히 축적되었으며,
- 기본 경로가 이미 안정적이어야 합니다.

그 이후에도 이 기능은 어디까지나 고급 모드여야지, 기본 경로가 되어서는 안 됩니다.

## 핵심 요약
- Preset은 관료주의가 아니라 재현성을 위한 인터페이스입니다.
- Raw entrypoint는 단기 유연성은 높지만 너무 많은 상태를 숨깁니다.
- 에이전트가 실험을 운영하는 ML 시스템에서는 무제한 자유보다 명시적 계약이 더 가치 있습니다.
- 사용자는 의미 있는 실험 설정을 바꾸고, 인프라 세부사항은 플랫폼이 떠안는 구조가 더 좋습니다.
