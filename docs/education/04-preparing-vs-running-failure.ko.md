# Preparing 실패와 Running 실패

## 왜 두 단계로 나누나?

후학습 런은 다양한 이유로 실패합니다. 일부 실패는 인프라와 관련된 것이고, 일부는 실험 자체와 관련된 것입니다. 이들을 한 데 묶으면 에이전트는 같은 스펙으로 재시도해야 하는지, 하이퍼파라미터를 바꿔야 하는지 판단할 수 없습니다.

`preparing` 상태는 이 경계선을 명시적으로 꺾으기 위해 존재합니다.

## Preparing 단계

`preparing`은 트레이너 프로세스가 시작되기 전에 발생하는 모든 것을 담습니다:

- Docker image pull
- Hugging Face Hub에서 베이스 모델 다운로드
- 데이터셋 다운로드 또는 로컬 경로 검증
- 캐시 워밍

이것들은 모두 **환경(environmental)** 단계입니다. 네트워크, 디스크 공간, 외부 서비스 가용성에 의존합니다. 러닝레이트나 LoRA rank와는 관련이 없습니다.

### 자주 발생하는 preparing 실패

| 실패 사유 | 무엇이 일어났는가 | 에이전트가 해야 할 일 |
|----------------|---------------|-----------------|
| `image_pull_failed` | Docker 레지스트리에 접근 불가 또는 잘못된 이미지 태그 | 나중에 재시도 또는 preset 이미지 확인 |
| `model_download_failed` | HF Hub 단일, 모델 ID 오타, 또는 디스크 부족 | 모델 ID 확인, 디스크 확인, 재시도 |
| `dataset_stage_failed` | 데이터셋을 찾을 수 없거나 split이 없거나 네트워크 오류 | 데이터셋 경로와 split 이름 확인 |

런이 `preparing`에서 실패하면 실험 자체는 무죄합니다. 에이전트는 환경이 양호할 때 정확히 같은 스펙으로 재제출할 수 있습니다.

## Running 단계

`running`은 트레이너 프로세스가 시작되는 순간 시작됩니다. 이 순간부터 실패은 실험이나 코드와 관련된 것입니다.

### 자주 발생하는 running 실패

| 실패 사유 | 무엇이 일어났는가 | 에이전트가 해야 할 일 |
|----------------|---------------|-----------------|
| `oom` | 배치 사이즈나 시퀀스 길이가 VRAM에 비해 큼 | `micro_batch_size`나 `max_seq_length` 감소 |
| `trainer_error` | 코드 예외, 데이터 포맷 오류, 또는 트레이너 버그 | `stderr.log` 확인 후 데이터 또는 설정 수정 |
| `timeout` | `resources.timeout`을 초과 | 타임아웃 증가 또는 `num_epochs` 감소 |
| `cancelled` | 에이전트나 사용자가 취소 신호 발송 | 밀린 로그 확인 후 재시도 여부 결정 |

이 실패들은 **실험적**입니다. 변경 없이 같은 스펙으로 재제출하면 또 실패할 가능성이 높습니다.

## 운영적 가치

이 단계들을 분리하면 에이전트에겐 명확한 결정 나무가 생깁니다:

1. `preparing`에서 실패했나?
   - 예 → 환경을 고치고 같은 스펙으로 재시도.
2. `running`에서 실패했나?
   - 예 → 스펙의 무엇인가를 바꾸고 재시도.

이 분리가 없다면 `oom`과 네트워크 방동이 똑같이 보입니다. 묶이지 않게 재시도하는 에이전트는 GPU 시간을 낭비하고 대장에 노이즈를 만듭니다.

## 실전 규칙

재시도를 결정하기 전에 항상 `failure_reason`을 확인하세요. "실패했으니까 다시 시도해보자"라고 만드시 마세요.
