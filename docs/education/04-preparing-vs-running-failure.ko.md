# Preparing 실패와 Running 실패를 왜 구분하는가

## 왜 이 경계가 필요한가
실패한 run이 모두 같은 성격의 실패는 아닙니다.

어떤 실패는 **트레이너가 시작되기 전에** 발생하고, 어떤 실패는 **학습이 이미 진행되는 중에** 발생합니다. 이 둘을 그냥 `failed` 하나로 뭉개 버리면 에이전트는 다음 중 무엇을 해야 할지 판단할 수 없습니다.
- 같은 스펙으로 재시도할지,
- 실험 설정을 바꿔야 할지,
- 아니면 환경을 고쳐야 할지.

바로 그래서 `preparing`과 `running`의 경계가 중요합니다.

## 상태 머신 관점
단순화한 run 생명주기는 대략 이렇게 볼 수 있습니다.

```text
queued -> preparing -> running -> succeeded
              |            |
              +-- failed   +-- failed / cancelled
```

핵심 질문은 하나입니다. **트레이너 프로세스가 실제로 시작됐는가?**

시작되지 않았다면 문제는 대개 환경 쪽입니다.
시작된 뒤라면 문제는 대개 실험 설정, 트레이너 코드, 자원 크기 산정 쪽입니다.

## `preparing`에 들어가는 일
`preparing`은 트레이너가 실제 작업을 시작하기 전에 해결되어야 하는 모든 단계를 포함합니다.

대표적인 예시는 다음과 같습니다.
- 컨테이너 이미지 pull,
- 베이스 모델 확인 또는 다운로드,
- 데이터셋 스테이징,
- 캐시 워밍,
- 마운트 경로 검증,
- 컨테이너/런타임 환경 생성.

이 단계들은 대부분 인프라 준비 상태와 관련됩니다.

## `running`에 들어가는 일
`running`은 트레이너 프로세스가 시작되는 순간부터입니다.

대표적인 예시는 다음과 같습니다.
- 학습 루프 시작,
- 배치 처리,
- loss 기록,
- 체크포인트나 어댑터 생성,
- 정상 종료 또는 오류 종료.

이 시점부터의 실패는 보통 실험 자체나, 실행 중 맞닥뜨린 자원 한계와 더 직접적으로 연결됩니다.

## 자주 보게 되는 `preparing` 실패
| 실패 사유 | 보통 의미하는 것 | 기본 대응 |
|---|---|---|
| `image_pull_failed` | 잘못된 이미지 참조, 레지스트리 장애, 인증 문제 | 이미지 확인 / 나중에 재시도 |
| `model_download_failed` | 잘못된 모델 ID, HF 장애, 디스크 문제 | ID/디스크 확인 후 재시도 |
| `dataset_stage_failed` | 경로 없음, split 없음, 스테이징 오류 | 데이터 소스 확인 |
| `container_create_failed` | 런타임 설정 오류, 마운트 문제, 디바이스 요청 문제 | 런타임 설정 점검 |

실전 규칙 하나는 이렇습니다. `preparing` 실패는 **스펙 자체는 여전히 유효할 수 있다**는 뜻인 경우가 많습니다.

## 자주 보게 되는 `running` 실패
| 실패 사유 | 보통 의미하는 것 | 기본 대응 |
|---|---|---|
| `oom` | 모델/배치/시퀀스가 VRAM보다 큼 | 메모리 압박 줄이기 |
| `trainer_error` | 트레이너 예외, 데이터 문제, 설정 버그 | 로그와 데이터 확인 |
| `timeout` | 시간 예산 초과 | 작업량 축소 또는 제한 상향 |
| `cancelled` | 사용자/에이전트가 run 중단 | 부분 결과물 점검 |

여기서도 규칙이 있습니다. `running` 실패는 **같은 스펙을 그대로 다시 넣으면 낭비일 가능성** 이 큽니다.

## 재시도 정책에 왜 중요한가
이 경계 덕분에 `failed`가 막연한 상태가 아니라 행동 가능한 진단으로 바뀝니다.

### `preparing`에서 실패했다면
진짜 문제는 환경일 수 있습니다.
예를 들면:
- 레지스트리 일시 장애,
- 일시적 네트워크 문제,
- 외부 모델 다운로드 실패.

이 경우에는 같은 스펙으로 다시 시도하는 것이 충분히 합리적일 수 있습니다.

### `running`에서 실패했다면
진짜 문제는 실험 설정이나 실행 예산일 수 있습니다.
예를 들면:
- 시퀀스 길이가 너무 김,
- 배치 크기가 너무 큼,
- 트레이너 버그,
- epoch가 너무 많아서 timeout 발생.

이 경우에는 같은 스펙을 그대로 재제출하는 것이 GPU 시간만 낭비할 가능성이 큽니다.

## 구체적인 예시
실패한 run 두 개를 가정해 봅시다.

- **Run A** 는 학습이 시작되기 전에 `model_download_failed`로 끝납니다.
- **Run B** 는 학습 시작 40초 뒤 `oom`으로 끝납니다.

둘 다 최종 상태는 `failed`지만, 다음 액션은 달라야 합니다.

- Run A: 환경을 점검한 뒤 같은 스펙으로 재시도
- Run B: 배치 크기, 시퀀스 길이, 모델 크기, timeout 중 하나를 조정한 뒤 재시도

이 차이를 표현하려고 상태 모델을 두 단계로 나누는 것입니다.

## 실패 대응 결정 트리
1. run이 `running`에 진입했는가?
   - 아니오 -> 이미지/모델/데이터셋/런타임 설정부터 본다.
   - 예 -> 트레이너 설정, 로그, 자원 크기 산정을 본다.
2. `failure_reason`은 무엇인가?
   - 환경/스테이징 사유 -> 같은 스펙이 여전히 유효할 수 있음
   - 실행/실험 사유 -> 먼저 무언가를 바꿔야 함
3. 일시적 외부 장애의 증거가 있는가?
   - 예 -> 같은 스펙 재시도 가능
   - 아니오 -> 스펙 수정 또는 코드 경로 수정 우선

## 단계별 디버깅 체크리스트
### `preparing`에서 실패했다면
- 이미지 참조가 정확하고 접근 가능한가?
- 모델 소스가 정상적이고 접근 가능한가?
- 데이터셋 경로나 split이 실제로 존재하는가?
- 스테이징과 캐시에 필요한 디스크 공간이 충분한가?
- 트레이너가 뜨기 전에 컨테이너 생성부터 실패한 것은 아닌가?

### `running`에서 실패했다면
- `stderr.log`에 예외가 기록되었는가?
- OOM으로 죽은 것은 아닌가?
- `resources.timeout`이 발동했는가?
- `micro_batch_size`, `max_seq_length`, `num_epochs`가 너무 공격적인가?
- 실패 전에 부분 결과물이라도 남았는가?

## 핵심 요약
- `preparing`과 `running`의 구분은 보기 좋은 상태 이름이 아니라 재시도 정책의 기준입니다.
- `preparing` 실패는 환경이나 스테이징 문제를 가리키는 경우가 많습니다.
- `running` 실패는 실험 설정, 코드, 자원 크기 문제를 가리키는 경우가 많습니다.
- 실패한 run을 무작정 다시 넣기 전에, 반드시 어느 단계에서 왜 실패했는지 확인해야 합니다.
